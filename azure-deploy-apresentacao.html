Deploy na Azure
com GitHub

Como hospedar uma aplicação estática usando Azure Static Web Apps com deploy automático via GitHub Actions.

Ana Flávia Lunedo  ·  Isabela Carbonera Branco

01 — Fundamentos

O que é Cloud Computing?

Recursos de TI entregues via internet, sob demanda, pagando pelo uso.

IaaS

Infraestrutura como serviço

Você gerencia SO e acima. Ex: Azure VMs.

PaaS

Plataforma como serviço

O provedor gerencia o SO. Você foca no código.

SaaS

Software como serviço

Produto pronto. Ex: Microsoft 365, GitHub.

Principais benefícios:

→

Escala automática conforme a demanda

→

Sem necessidade de hardware próprio

→

Alta disponibilidade global

→

Segurança e atualizações gerenciadas

02 — Serviço utilizado

Azure Static Web Apps

Serviço PaaS da Microsoft para hospedar sites estáticos com CI/CD nativo via GitHub.

→

Plano gratuito

100 GB de banda/mês, SSL gerenciado, domínio personalizado.

→

CDN global

Conteúdo distribuído em pontos ao redor do mundo, com baixa latência.

→

Integração nativa com GitHub

Ao criar o recurso, o workflow de deploy é gerado automaticamente.

→

Preview por Pull Request

Cada PR abre um ambiente de staging automático.

Fluxo de dados:

💻 Computador git push
🐙 GitHub Actions + CI/CD
☁ Azure CDN hospedagem global
🌍 Usuário final qualquer lugar
03 — Antes de começar

Pré-requisitos

O que você precisa ter antes de fazer o deploy.

01

Conta no GitHub

Gratuita em github.com. O repositório pode ser público ou privado.

02

Conta na Azure

Cadastro gratuito em azure.microsoft.com, com crédito inicial de US$ 200.

03

Projeto no repositório

Arquivos HTML/CSS/JS já commitados e disponíveis no GitHub.

04

Git instalado

Para enviar alterações do computador ao repositório. Verifique com git --version.

04 — Passo a passo

Criando o recurso na Azure

01

Acesse portal.azure.com e faça login

Clique em "Criar um recurso" (+) no menu lateral.

02

Pesquise "Static Web Apps"

Selecione o serviço da Microsoft e clique em "Criar".

03

Configure o recurso

Defina assinatura, grupo de recursos, nome do app e região. Plano: Gratuito.

04

Conecte ao GitHub

Autorize a Azure. Selecione organização, repositório e branch (main).

05

Revisar e criar

A Azure gera o arquivo de workflow no seu repositório e faz o primeiro deploy automaticamente.

06 — Verificação

Acompanhando o deploy

Como saber se deu certo após o push.

①

GitHub → aba Actions

Veja o workflow rodando em tempo real, com logs por step.

②

Portal Azure → Visão geral

Status do deploy e URL pública da aplicação.

③

URL gerada

https://seu-app.azurestaticapps.net

Status possíveis no Actions:

⏳

Queued — aguardando runner

🔄

In progress — executando steps

✓

Success — deploy concluído

✕

Failure — verifique o log do step com erro

Em caso de falha, clique no job com ✕ dentro de Actions para ver o log detalhado.
07 — Boas práticas

O que fazer (e não fazer)

Hábitos simples que evitam dores de cabeça.

→

Use branches por ambiente

main para produção, dev para desenvolvimento. Nunca suba código não testado direto na main.

→

Escreva commits descritivos

Use prefixos como feat:, fix:, docs: para manter o histórico legível.

→

Nunca commite segredos

API keys e tokens sempre em GitHub Secrets. Adicione .env ao .gitignore.

→

Use Pull Requests

O Static Web Apps cria um ambiente de preview automático para cada PR — use para revisar antes de publicar.

08 — Resumo

O que aprendemos

Três ideias para levar desta apresentação.

1

Cloud elimina a gestão de servidores

A Azure cuida de disponibilidade, segurança e escala — você foca no produto.

2

Static Web Apps é gratuito e simples

CDN global, HTTPS automático, deploy nativo via GitHub — tudo no plano free.

3

CI/CD automatiza o trabalho repetitivo

Cada git push na main dispara o pipeline. Do commit ao ar em menos de 3 minutos.
